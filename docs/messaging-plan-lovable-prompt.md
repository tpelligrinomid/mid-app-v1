# Messaging Plan Deliverable — Lovable Frontend Build Prompt

## Overview

Add a new deliverable type, **Messaging Plan**, to the Compass Deliverables module. It is a StoryBrand-based brand messaging document, generated by AI and rendered as **markdown**.

The backend is fully built and deployed. A Messaging Plan behaves like the existing **Research** deliverable: AI-generated, markdown output, no custom structured renderer. The only meaningful new UI is making the type selectable and giving it a generate form (which mirrors the existing Content Plan generate modal).

---

## Backend API

**Base URL:** `https://mid-app-v1.onrender.com`

**Auth:** All requests require the Supabase JWT:
```typescript
const response = await fetch('https://mid-app-v1.onrender.com/api/compass/deliverables/...', {
  headers: {
    'Authorization': `Bearer ${session.access_token}`,
    'Content-Type': 'application/json'
  }
});
```

The deliverable type slug is **`messaging_plan`**.

---

## 1. Register the type

Wherever the app lists deliverable types (the "create deliverable" type picker, type labels/badges, icons, filters), add:

- **Value:** `messaging_plan`
- **Display label:** `Messaging Plan`
- **Icon/badge:** reuse the same treatment as other strategy deliverables (research / roadmap / plans).

It is an **AI-generatable** type (it should show the "Generate with AI" affordance, like research/roadmap/content_plan — not the manual-only types).

---

## 2. Create the deliverable

No change to the create flow — use the existing endpoint. Just ensure `messaging_plan` is an allowed `deliverable_type`.

**API:** `POST /api/compass/deliverables`

```json
{
  "contract_id": "uuid",
  "title": "Acme Corp — Messaging Plan",
  "deliverable_type": "messaging_plan",
  "status": "planned"
}
```

---

## 3. Generate form (the main new UI)

When the user clicks **Generate with AI** on a `messaging_plan` deliverable, show a modal **identical in structure to the existing Content Plan generate modal**:

### Fields

1. **Info banner** (purple, like the content plan one):
   > Messaging plans work best when **research** has been completed for this contract. The AI automatically pulls context from the research report and roadmap (if available).

2. **Key meetings to prioritize** — a selectable list of the contract's meetings that have transcripts (multi-select checkboxes). Same component the content plan modal uses. Label: *"Select the brand-story, kickoff, or planning meeting with the client."* These become `primary_meeting_ids`.

3. **Instructions for AI (optional)** — free-text textarea. Placeholder: *"e.g. Emphasize the new enterprise positioning, lead with the founder's POV."*

There is **no** client/competitor input on this form — the AI derives the client from the existing research deliverable. (This matches the content plan modal, which also doesn't collect client info.)

### Submit

**API:** `POST /api/compass/deliverables/{deliverable_id}/generate`

```json
{
  "primary_meeting_ids": ["uuid", "uuid"],
  "instructions": "Optional guidance text"
}
```

**Response:** `202 Accepted`
```json
{
  "message": "Generation started",
  "deliverable_id": "uuid",
  "processing": { "status": "pending" }
}
```

Roadmap is **not** required — unlike ABM/content plans, the Messaging Plan generate call will never return a `422 MISSING_ROADMAP`. Don't gate the button on a roadmap existing.

---

## 4. Generation status (polling)

Reuse the existing generation-status polling exactly as research/roadmap do.

**API:** `GET /api/compass/deliverables/{deliverable_id}/generation-status`

```json
{
  "generation": {
    "status": "assembling_context | submitted | completed | failed",
    "job_id": "...",
    "submitted_at": "...",
    "completed_at": "...",
    "error": "present only on failure"
  }
}
```

Show the same "working / generating" state in the deliverable UI that other AI deliverables show. When `status` becomes `completed`, refetch the deliverable and render its content.

---

## 5. Render the result

The generated output lands in **`content_raw` as markdown** (`content_structured` will be `null`).

**API:** `GET /api/compass/deliverables/{deliverable_id}` → `deliverable.content_raw`

**Reuse the existing Research deliverable renderer** — render `content_raw` as markdown. No custom section components, no structured-data view. The markdown contains all sections (Market Opportunity, Positioning, Brand Story, Pillars, ICP Messaging, Products matrix as a markdown table, Use/Avoid lists), so a standard markdown renderer with table support is all that's needed.

Keep the existing actions available on the rendered deliverable (edit, copy, version, ClickUp link, etc.) — same as research.

---

## Summary

| Step | Work |
|---|---|
| Type registry | Add `messaging_plan` → "Messaging Plan", mark as AI-generatable |
| Create | No change (type just needs to be allowed) |
| Generate modal | Clone the Content Plan modal: meeting multi-select + optional instructions; no client fields; no roadmap gating |
| Status | Reuse existing `generation-status` polling |
| Render | Reuse the Research markdown renderer on `content_raw` |

Everything except the generate modal is reuse — the heavy lifting is registering the type and wiring the (mostly identical) generate form.
